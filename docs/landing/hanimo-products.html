03Multi-Terminal · Multi-Session Workflow

hanimo TUI는 *단일 바이너리*라서 터미널 여러 개 띄우면 자동으로 N개의 독립 세션이 된다. 각 세션은 자기 SQLite DB / 자기 모드 / 자기 모델 / 자기 프로젝트 디렉토리를 갖고, .hanimo.md + .hanimo/knowledge/로 컨텍스트가 자동 분리된다.

① 세션 격리 원칙

각 hanimo 인스턴스는 *현재 작업 디렉토리*를 root로 잡는다. 다른 프로젝트면 다른 컨텍스트, 다른 세션 DB. 같은 프로젝트면 자동 공유.

② 모델 분담

무거운 모델은 1~2개 세션, 빠른 모델은 N개 세션에 분산. /model로 세션별 즉시 전환. BYOK라 사용자 결제 키 1개를 N 세션이 공유해도 OK.

③ 역할 분담

Planner / Coder / Reviewer / Tester / Doc-writer로 각 세션을 나누면 컨텍스트 오염 없이 병렬. 각 세션 시작 시 역할 프롬프트 1개만 붙여넣으면 됨 (아래 §4 템플릿).

④ 동기화

세션 간 결과 공유는 *파일*로 한다 (Reviewer가 Coder의 출력을 읽음). 직접 메시지 패싱 안 함 — 단순함이 곧 안정성. 필요하면 Plan 모드의 plan.json 활용.

tmux 4-pane 셋업 — 한 명령으로 시작

프로젝트 루트에 두면 좋은 scripts/hanimo-tmux.sh 예시. 한 번 실행하면 Planner / Coder / Reviewer / Run 4 pane이 같은 프로젝트 디렉토리를 공유하며 뜬다. 

#!/usr/bin/env bash
# scripts/hanimo-tmux.sh — 한 프로젝트에 hanimo 4-pane 띄우기
# Usage: ./scripts/hanimo-tmux.sh [session-name]

set -euo pipefail
SESSION=${1:-hanimo}
ROOT=$(pwd)

tmux new-session -d -s "$SESSION" -c "$ROOT" -n main

# pane 1 — Planner (Plan mode, 큰 모델)
tmux send-keys -t "$SESSION":main "hanimo --mode plan" C-m

# pane 2 — Coder (Dev mode, 자율 200회)
tmux split-window -h -t "$SESSION":main -c "$ROOT"
tmux send-keys -t "$SESSION":main.2 "hanimo --mode dev" C-m

# pane 3 — Reviewer (Super mode, 읽기 위주)
tmux split-window -v -t "$SESSION":main.1 -c "$ROOT"
tmux send-keys -t "$SESSION":main.3 "hanimo" C-m

# pane 4 — Run/Test (일반 셸)
tmux split-window -v -t "$SESSION":main.2 -c "$ROOT"

tmux select-pane -t "$SESSION":main.1
tmux attach -t "$SESSION"

위 스크립트는 4 pane을 띄우지만 각 pane은 자기 hanimo 인스턴스이므로 SQLite 세션도 4개로 분리. 단, 같은 프로젝트 디렉토리라 .hanimo.md + ~/.hanimo/sessions.db의 메모리 / 사용량 통계는 공유.

04Prompt Templates per Session

각 pane을 띄운 직후 *첫 메시지*로 붙여넣을 역할 부여 프롬프트 4종. 이걸 한 번 넣으면 그 세션은 정확히 그 역할만 수행하며, 다른 pane 결과를 *파일*로만 참조한다.

Pane 1 · Planner

요구사항을 단계별 계획서로 변환

hanimo --mode plan으로 띄워진 세션. 사용자 요구를 받아 실행 단계 JSON을 생성, /approve 후 단계별 명령을 발행한다. 

너는 Planner 세션이다. 코드는 직접 수정하지 마라.
다음 규칙을 지켜라:

1. 사용자 요구를 받으면 먼저 .plan.md로 단계 목록을 출력한다.
2. 각 단계는 다음 형식: "[N] (Coder|Reviewer) -- 작업 한 줄".
3. 추정·가정이 있으면 단계 위에 "ASSUME: ..."로 명시.
4. .plan.md 작성 후 사용자 승인을 기다린다.
5. 승인 후 .plan.md의 [1]부터 *Coder pane에 줄 명령*만 출력한다.
6. 직접 file_write를 호출하지 않는다 — 그건 Coder의 일.

승인 받기 전에는 어떤 파일도 만들지 말 것.
Pane 2 · Coder

Plan의 단계만 정확히 실행

hanimo --mode dev (자율 200회). Planner가 발행한 단계 하나를 받아 끝까지 구현. 끝나면 [STEP_COMPLETE]로 알린다. 

너는 Coder 세션이다. 너의 일:

1. 메시지로 "[N] -- 작업" 형태가 들어오면 그 단계만 실행한다.
2. 그 단계와 무관한 코드는 절대 건드리지 않는다.
3. 끝나면 마지막에 [STEP_COMPLETE: N] 한 줄로 보고한다.
4. 막히면 [STEP_BLOCKED: N -- 이유] + 질문을 출력하고 멈춘다.
5. .plan.md는 읽기 전용. 절대 수정하지 마라.
6. Reviewer pane이 코드 검토 중일 가능성이 있다 — 빌드를 깨면 안 된다.

매 단계마다 자동으로 go test / npm run build 등
관련 검증을 실행하고 그 결과도 보고에 포함한다.
Pane 3 · Reviewer

읽기 전용으로 Coder 출력 검증

Super 모드 (기본 모드). 파일 수정 권한 없이 file_read / git_diff / grep_search만 사용하여 코드 리뷰 노트를 .review.md로 출력. 

너는 Reviewer 세션이다. 절대 file_write / file_edit 호출 금지.
허용 도구: file_read, list_files, grep_search, glob_search, git_diff,
       git_log, diagnostics.

작업 흐름:

1. 사용자가 "review STEP_N" 형태로 트리거하면,
2. git diff로 직전 변경분만 본다.
3. 다음 항목을 체크: 의도 부합 / 회귀 / 보안 / 가독성 / 테스트.
4. 결과를 .review-N.md에 *추가*한다 (덮어쓰지 마라).
5. 마지막에 [REVIEW_OK: N] 또는
   [REVIEW_REJECT: N -- 사유]로 1-line 보고.

추측은 금지. 실제 본 파일·라인 번호만 인용한다.
Pane 4 · Run / Test

실제 빌드·실행은 hanimo 밖 일반 셸

hanimo가 아닌 *일반 셸*. go test ./... / npm run dev / make build-onprem 등을 직접 돌리고 출력을 다른 pane에서 file_read로 회수. 

# 이 pane은 hanimo가 아닌 일반 셸. 자동화 없음.
# 다른 pane(Reviewer/Coder)이 검증을 요청하면 여기서 직접 실행.

# 권장 패턴 — 출력을 파일로 떨어뜨려 다른 pane이 읽게 함
go test ./... 2>&1 | tee .test-output.log
make build-onprem 2>&1 | tee .build-output.log
npm run dev > .dev-server.log 2>&1 &

# 실패 시 마지막 30줄 빠르게 보기
tail -30 .test-output.log

파일로 통신하는 이유

방식장점단점
파일 매개 (.plan.md / .review-N.md / .test-output.log) 각 pane은 자기 SQLite 세션만 신경. 컨텍스트 오염 없음. git diff로 추적 가능 약간 verbose
SSE / IPC 직접 메시징 실시간 인스턴스 간 결합 ↑, 디버깅 어려움. 한 세션 죽으면 전체 멈춤
단일 Plan 모드 + Auto 한 세션 compaction 빨리 발동, 200회 한계, 역할 구분 X
팁: 4 pane 모두 같은 프로젝트 디렉토리에 있으므로 .hanimo/knowledge/의 프로젝트 지식 문서 + .hanimo.md 프로젝트 프로파일이 모든 pane에 자동 공급된다. 즉 *"역할 부여"* 프롬프트만 pane별로 다르고, 도메인 지식은 자동 공유.
