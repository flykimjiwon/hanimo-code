0112-Layer Architecture

사용자 키 입력부터 LLM 스트리밍·도구 실행·UI 렌더링·세션 영속화까지 hanimo TUI의 단일 프로세스 안에서 일어나는 흐름을 12 레이어로 분해. 각 레이어는 README.md "Internal Process Flow"에서 도식 풀버전 확인.

사용자 (터미널) ↓ 키보드 입력 [Bubble Tea 이벤트 루프] ← Layer 1 ↓ msg dispatch [키 입력 → sendMessage] ← Layer 2 ↓ [LLM 스트리밍 + 도구 루프] ← Layer 3 ↓ [View() 렌더링] ← Layer 4 ↓ [패키지 책임 맵] ← Layer 5 ↓ [/auto 자율 모드] ← Layer 6 [Plan: 계획→승인→실행] ← Layer 7 [3-Stage Compaction] ← Layer 8 [Knowledge 주입 파이프] ← Layer 9 [위험 명령 확인 플로우] ← Layer 10 [SQLite 세션 영속화] ← Layer 11 [멀티에이전트 오케스트레이션] ← Layer 12
01

Bubble Tea 이벤트 루프

internal/app/app.go — Update(msg)

모든 입력·타이머·스트리밍 청크가 단일 msg 채널로 들어와 순차 처리. UI 갱신 깜빡임을 막기 위해 150ms 틱으로 분리.

tea.KeyPressMsg tea.PasteMsg streamTickMsg streamChunkMsg toolResultMsg
02

키 입력 → 메시지 전송 파이프라인

handleSlashCommand() · sendMessage()

오버레이(setup/palette/menu/askUser/dangerCmd)가 활성이면 일반 키 처리 스킵. 슬래시(/)는 명령 디스패처로, 그 외엔 LLM 메시지로.

Tab → 모드 전환 Ctrl+K → 팔레트 ↑/↓ → 입력 히스토리
03

LLM 스트리밍 + 도구 실행 루프

internal/llm/client.go — StreamChat()

OpenAI 호환 SSE 청크를 streamCh 채널로. tool_calls 도착 시 도구 실행 → history 추가 → 재요청. 최대 20회(/auto는 200회).

SSE streaming tool loop ≤20 drift detection danger gate
04

View() 렌더링 스택

internal/ui/ — chat.go · styles.go · menu.go · palette.go

viewport(스크롤) → intent hint → ASK_USER 블록 → textarea → 상태바 → 오버레이 순으로 위에서 아래로 합성.

5 themes i18n ko/en Lip Gloss
05

파일별 책임 맵

"이 부분 고쳐"의 진입점 인덱스

입력/이벤트는 app.go, 도구는 internal/tools/registry.go, 프롬프트는 internal/llm/prompt.go, Git HUD는 internal/gitinfo/. README.md Layer 5에 풀 인덱스.

06

/auto 자율 모드 루프

internal/agents/auto.go — CheckAutoMarkers()

시스템 프롬프트에 [AUTO_COMPLETE]/[AUTO_PAUSE] 마커 주입. 매 스트림 끝에 마커 검사. 미발견 시 200회까지 자동 재요청.

07

Plan 모드 — 계획 → 승인 → 단계별 실행

internal/agents/plan.go — ParsePlan()

JSON plan 생성 → 사용자 /approve → 단계마다 [STEP_COMPLETE]/[STEP_FAILED] 마커. 단계 간 컨텍스트 분리.

08

3-Stage Compaction

internal/llm/compaction.go — Compact()

Snip (40+ msgs, tool 결과 절단) → Micro (4000자+ 메시지 자르기) → LLM 요약 (ctx ≥90% 자동, /compact 수동).

ctx<70% normal 70-79% warn 80-89% critical ≥90% auto
09

Knowledge 주입 파이프라인

internal/knowledge/ — Inject()

키워드 매칭 → BM25 스코어링 → 토큰 예산 8192 내 Tier 우선 선택. 소스 3곳: knowledge/docs/(62개 임베디드) + .hanimo/knowledge/(프로젝트) + ~/.hanimo/knowledge/(글로벌).

10

위험 명령 확인 플로우

internal/tools/shell.go — IsDangerous()

shell_exec 호출 시 dangerous regex 매칭(rm -rf, sudo, git push --force, dd if=, chmod 777, curl|sh 등). 위험 감지 시 Allow/Deny 오버레이로 사용자 승인 강제.

11

SQLite 세션 영속화

internal/session/ — db/schema.sql

매 메시지(user/assistant/tool)마다 ~/.hanimo/sessions.db 자동 저장. /save 이름 지정, /load 복원, /search 전문검색, /export 마크다운.

sessions messages memories usage_log mcp_servers
12

멀티에이전트 오케스트레이션

internal/multi/ — orchestrator.go · strategy.go · merge.go

Agent1(Super 모델) + Agent2(Dev 모델) 병렬 실행. Review(생성→리뷰), Consensus(독립→합성), Scan(파일 파티셔닝), Auto(키워드 자동선택).

04Modes & Slash Commands

Tab으로 순환되는 3 모드 + 30+ 슬래시 명령. 각 모드는 시스템 프롬프트와 허용 도구 집합이 다르다.

Mode 1 / Tab

Super

만능. 모든 도구 + clarify-first directive. 일반 사용 기본.

Mode 2 / Tab Tab

Dev (Deep Agent)

자율 100~200회. ASK_USER 최소화. 무거운 코딩 작업.

Mode 3 / Tab Tab Tab

Plan

JSON 계획 → 승인 → 단계 실행. 분석·아키텍처 검토.

05Extension System — Knowledge · Skills · MCP

hanimo는 *임베디드 지식*과 *외부 확장*을 동시에 지원한다. 사용자가 새 문서를 떨구거나 MCP 서버를 연결하면 시스템 프롬프트와 도구 목록에 동적으로 합쳐진다.

Knowledge Sources (3 layers)

소스위치로드 시점
임베디드 (62 docs) knowledge/docs/ — go:embed 빌드 시점에 박힘 · 인터넷 불요
프로젝트 .hanimo/knowledge/*.md 해당 폴더에서 hanimo 실행 시 자동 인덱싱
글로벌 ~/.hanimo/knowledge/*.md 모든 hanimo 인스턴스 공유

Skills System

.hanimo/skills/<name>/SKILL.md 형식. agentskills.io 표준 호환. 프론트매터 + $ARGUMENTS + 인라인 !shell 명령 + lazy load 지원. 

---
name: review-pr
description: PR 리뷰 워크플로
trigger: review-pr $PR_NUMBER
---

다음 PR을 리뷰해라:
- !gh pr view $PR_NUMBER --json title,body
- 변경분 분석
- ...

MCP — Model Context Protocol

JSON-RPC 2.0 over stdio 또는 SSE. ~/.hanimo/config.yaml의 mcp_servers에 정의. 외부 서버의 도구가 hanimo의 도구 목록에 자동 합쳐짐. desktop IDE에서는 라이브 트리 뷰 (Phase 12). 

mcp_servers:
  - name: filesystem
    transport: stdio
    command: npx -y @modelcontextprotocol/server-filesystem ~/Desktop
  - name: github
    transport: sse
    url: http://localhost:3001/sse
한국 MCP 카탈로그 35종. Phase 19에서 한국 시장에 특화된 MCP 서버 35개를 카탈로그로 박아놨다. 사용자가 /mcp로 토글만 하면 즉시 사용. wedge: "한국 폐쇄망/공공/금융 + 한국어" 시장에 hanimo가 자연 정렬.

06Multi-Agent Orchestration

듀얼 모델로 같은 작업을 *역할 나눠* 동시 진행. Agent1(Super 모델, 쓰기 권한) + Agent2(Dev 모델, 보조). 4 전략을 /multi로 토글.

전략흐름적합한 작업
/multi review Agent1 생성 → Agent2 읽기전용 검토 → LLM 합성 코드 변경 + 회귀 점검
/multi consensus Agent1·2 독립 실행 → LLM이 두 응답 비교·합성 설계 결정 / 비교 분석
/multi scan 파일/함수 단위 파티셔닝 → 영역별 담당 → 결과 합산 대규모 리팩토링 / 검색
/multi auto 입력 키워드로 위 3 전략 자동 선택 무엇을 쓸지 모를 때

구현: internal/multi/orchestrator.go · strategy.go · merge.go. 각 에이전트 최대 20 tool iterations · 실시간 progress가 companion SSE로 송출.

07Build Profiles (4)

같은 코드베이스 + 다른 빌드 플래그 = 다른 배포 채널. internal/config ldflags + internal/build 빌드 태그가 분기.

프로파일명령대상특징
vanilla make build OSS 사용자 아무것도 baked 안 됨. 사용자가 endpoint/key 설정.
distro make build-distro 회사 배포 endpoint/provider/model frozen, API_KEY는 사용자가 런타임 입력
sealed make build-sealed 내부 단일 배포 endpoint + API_KEY 모두 frozen. 비공개 유지 필수.
onprem (LTS) make build-onprem 폐쇄망 / 분기 LTS -tags=onprem으로 컴파일 타임 텔레메트리 0 보장. 분기 1회 발행.

크로스 플랫폼

각 프로파일별 build-*-all 타겟이 5 OS×Arch 매트릭스를 한 번에 빌드: macOS arm64/amd64 · Linux amd64/arm64 · Windows amd64. 

# 예시: 분기 LTS 풀 매트릭스
make build-onprem-all
# dist/hanimo-onprem-darwin-arm64
# dist/hanimo-onprem-darwin-amd64
# dist/hanimo-onprem-linux-amd64
# dist/hanimo-onprem-linux-arm64
# dist/hanimo-onprem-windows-amd64.exe
desktop 빌드는 별도. hanimo-code-desktop/에서 wails build · macOS .app · Windows .exe · Linux는 GH Actions에서 빌드 (Phase 18).
