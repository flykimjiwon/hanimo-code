# Hanimo 플랫폼 전략 가이드

> 터미널 도구 / TUI / 데스크톱 앱 / VSCode 익스텐션 — 어디에 투자할 것인가

---

## 현재 상황

### hanimo가 가진 것

| 플랫폼 | 상태 | 기술 |
|--------|------|------|
| Terminal (text-mode) | ✅ 완성 | readline REPL |
| TUI (fullscreen) | ✅ 완성 | Ink/React |
| Desktop App | 🔨 개발 중 | Tauri v2 + Web React |
| VSCode Extension | 📋 계획 | webview |

### 경쟁 환경

| 제품 | 형태 | 성공 여부 | 핵심 전략 |
|------|------|-----------|----------|
| Claude Code | 터미널 도구 | 대성공 | 개발자가 이미 있는 곳 (터미널) |
| Cursor | IDE (VSCode 포크) | 대성공 | 개발자가 이미 있는 곳 (에디터) |
| Windsurf | IDE (VSCode 포크) | 성공 | Cursor 대안 |
| Aider | 터미널 도구 | 성공 | 경량, 오픈소스 |
| Warp | AI 터미널 앱 | 성장 중 | 터미널 자체를 재발명 |
| Continue.dev | VSCode 익스텐션 | 성장 중 | 프로바이더 독립적 |

**공통점: 성공한 제품은 모두 "개발자가 이미 있는 곳"에 존재한다.**

---

## 핵심 질문: 사용자는 어디에 있는가?

```
개발자의 하루:

  터미널 (40%)  ←── hanimo가 여기에 있음 ✅
       ↕
  에디터 (50%)  ←── VSCode 익스텐션이 여기를 노림
       ↕
  별도 앱 (5%)  ←── 데스크톱 앱은 여기... 🤔
       ↕
  브라우저 (5%)
```

**코딩 에이전트는 코드가 있는 곳에서 써야 한다.**
별도 앱을 열면 에디터↔앱 컨텍스트 스위칭이 발생하고, 파일 트리/에디터 통합이 없으면 기존 도구 대비 약점만 부각된다.

## 플랫폼별 상세 분석

### 1. 터미널 도구 (현재 핵심)

**투자 대비 효과: ★★★★★**

```
장점:
  ✅ 이미 완성됨
  ✅ 개발자가 매일 쓰는 환경
  ✅ SSH/원격 서버에서도 동작
  ✅ CI/CD 파이프라인 통합 가능
  ✅ Claude Code가 시장 검증 완료
  ✅ 설치 간단 (npm install -g)

약점:
  ❌ 비개발자 접근성 낮음
  ❌ 시각적 제한 (diff 뷰, 이미지 등)

결론: 핵심 제품. 계속 투자.
```

### 2. VSCode 익스텐션

**투자 대비 효과: ★★★★☆**

```
장점:
  ✅ 개발자의 80%가 VSCode 사용
  ✅ 에디터 통합 (파일 트리, diff, 인라인 수정)
  ✅ Marketplace 배포 → 발견 가능성 높음
  ✅ Continue.dev가 "프로바이더 독립적 익스텐션" 시장 검증
  ✅ hanimo의 차별점 (14개 프로바이더) 유지 가능
  ✅ core/ 엔진 재사용 가능

약점:
  ❌ VSCode API 학습 곡선
  ❌ Cursor/Windsurf와 직접 경쟁
  ❌ VSCode 생태계에 종속

구현 난이도: 중 (core 재사용, UI는 webview)
예상 작업량: 2-3주
```

### 3. 데스크톱 앱 (Tauri)

**투자 대비 효과: ★★☆☆☆ (일반 앱) → ★★★☆☆ (AI 터미널 컨셉)**

```
일반 앱 (채팅 + 에디터 + 파일트리) 문제:
  ❌ "왜 Cursor 대신 이걸 써야 하지?" 에 답이 없음
  ❌ 에디터 품질이 VSCode/Cursor를 절대 못 따라감
  ❌ 파일 시스템 통합이 네이티브 IDE보다 약함
  ❌ 만들어도 쓸 사람이 명확하지 않음

AI 터미널 앱 (Warp 스타일) 가능성:
  ✅ 터미널인데 예쁘고 AI가 내장
  ✅ 비개발자도 접근 가능한 인터페이스
  ✅ 세션 관리, 비용 대시보드 등 부가 기능
  ✅ 터미널 도구의 강점 유지 + GUI의 접근성
  ✅ Warp가 시장 검증 중

위험:
  ⚠ 터미널 에뮬레이터 구현 난이도 높음 (xterm.js)
  ⚠ Warp와 직접 경쟁
  ⚠ 개발 기간 길어질 수 있음
```

## 추천 전략

### 우선순위

```
┌─────────────────────────────────────────────────────┐
│  Phase 1: 터미널 도구 완성도 (지금)                  │
│  ├─ 스킬 시스템 구현                                │
│  ├─ agent-loop 안정화 (retry, 토큰 컴팩션)          │
│  └─ core/ 엔진 UI 독립적 분리                       │
│                                                     │
│  Phase 2: VSCode 익스텐션 (다음)                     │
│  ├─ core/ 엔진을 익스텐션에서 호출                   │
│  ├─ 채팅 패널 (webview)                             │
│  ├─ 인라인 코드 수정 통합                            │
│  └─ Marketplace 배포                                │
│                                                     │
│  Phase 3: 데스크톱 앱 — "AI 터미널" (선택)           │
│  ├─ Tauri + xterm.js 터미널 에뮬레이터              │
│  ├─ hanimo TUI를 앱 내 터미널에서 실행              │
│  ├─ 세션 관리 / 비용 대시보드 / 프로바이더 설정 UI  │
│  └─ 비개발자 타겟 온보딩                            │
└─────────────────────────────────────────────────────┘
```

### Phase 1이 최우선인 이유

모든 플랫폼이 `core/` 엔진을 공유합니다. core가 탄탄해야 어디에 올려도 품질이 보장됩니다:

```
                    ┌─────────────┐
                    │   core/     │  ← 이게 튼튼해야 전부 튼튼
                    │  agent-loop │
                    │  tools      │
                    │  providers  │
                    │  skills     │
                    └──────┬──────┘
                           │
         ┌─────────┬───────┼────────┬──────────┐
         │         │       │        │          │
      Terminal    TUI    App     VSCode     CLI/CI
```

│  │ 파일   │ │                  │ │
│  │ 트리   │ │  Monaco 에디터   │ │
│  │        │ │                  │ │
│  │        │ │                  │ │
│  │        │ ├──────────────────┤ │
│  │        │ │  채팅 패널       │ │
│  └────────┘ └──────────────────┘ │

문제:
- Cursor가 이미 완벽하게 함

## 각 플랫폼별 구현 전략

### 터미널 도구 (이미 완성)

```
유지보수 + 개선:
- 스킬 시스템 구현 (설계 완료)
- retry/backoff 추가
- 토큰 기반 컴팩션
- TUI 세션 resume 연결
```

### VSCode 익스텐션

```
구현 계획:
1. core/ 엔진을 Node.js 프로세스로 실행
2. VSCode Extension API로 통신 (Language Server Protocol 또는 custom)
3. 채팅 패널: webview (React)
4. 인라인 수정: VSCode TextEdit API
5. 파일 변경 감지: FileSystemWatcher
6. 상태바: 토큰/비용/모델 표시

재사용 가능한 것:
- core/agent-loop.ts → 그대로
- core/tools/* → 그대로
- providers/* → 그대로
- config/* → 그대로
- headless.ts → 통신 프로토콜 기반

새로 만들어야 하는 것:
- VSCode Extension Host 코드
- Webview React UI
- TextEdit 통합
- Extension 설정 UI
```

### 데스크톱 앱 (AI 터미널 컨셉)

```
구현 계획:
1. Tauri v2 + xterm.js (터미널 에뮬레이터)
2. hanimo CLI를 앱 내 터미널에서 실행
3. 사이드바: 세션 목록, 비용 대시보드
4. 설정: GUI로 프로바이더/모델 설정
5. MCP 서버 관리 UI

재사용 가능한 것:
- hanimo CLI 전체 (앱 내 터미널에서 그대로 실행)
- 설정 파일 (~/.hanimo/config.json)

새로 만들어야 하는 것:
- xterm.js 터미널 통합
- 세션 관리 UI
- 비용 대시보드
- 프로바이더 설정 GUI
- 온보딩 위저드 (GUI 버전)
```

## 현재 Tauri 앱 작업에 대한 제안

이미 진행 중인 Tauri 앱 작업이 있으므로:

### 단기 (지금)
1. **앱 완성은 계속 진행** — 학습 + 포트폴리오 가치가 있음
2. 단, **VSCode 클론 방향이면 재고** → AI 터미널 방향으로 전환 검토
3. core/ 분리를 의식하면서 작업 — 앱 전용 로직과 공통 로직 구분

### 중기 (앱 1차 완성 후)
1. **터미널 도구 안정화** — 스킬 시스템, retry, 테스트
2. **VSCode 익스텐션 착수** — 실사용자 확보 효과 가장 큼

### 장기
1. 3개 플랫폼(터미널/앱/익스텐션) 동시 지원
2. core/ 엔진 공유로 기능 동기화
3. 각 플랫폼별 고유 강점 강화

## 핵심 원칙

1. **core가 핵심이다** — UI는 바꿀 수 있지만, core 엔진의 품질이 제품 가치를 결정한다
2. **사용자가 있는 곳에 가라** — 터미널(개발자) > VSCode(개발자) > 독립 앱(?)
3. **경량을 유지하라** — hanimo의 차별점은 7,800줄의 효율성. 무거워지면 경쟁력 상실
4. **하나를 잘 만들고 확장하라** — 터미널 도구를 완벽히 → 그 위에 다른 플랫폼 확장
